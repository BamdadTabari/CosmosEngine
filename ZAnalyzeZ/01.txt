# Cosmos Engine Architecture Document

## 1. Repository Structure

Top-level layout:

- `Cosmos.Domain/`
  - Core domain model
- `Cosmos.Engine/`
  - Physics, orbital mechanics, maneuvers, analysis
- `Cosmos.Desktop/`
  - Visualization, runtime loop, input, rendering, data loading
- `Cosmos.Engine.Tests/`
  - Placeholder test project
- `Docs/`
  - Project documentation and research notes
- `CosmosEngine.slnx`
  - Solution file
- `README.md`
  - High-level project description

## 2. Project Responsibilities

### Cosmos.Domain
Purpose: define the domain vocabulary for celestial simulation.

Responsibilities:
- represent bodies, universe contents, and physical state
- define basic value objects, vectors, enums, and navigation abstractions
- remain independent of engine implementation and rendering

### Cosmos.Engine
Purpose: provide the simulation engine and orbital mechanics services.

Responsibilities:
- implement gravity-based physics stepping
- encapsulate integration algorithm
- compute orbital metrics and maneuvers
- model navigation solution generation
- track orbit extrema and influence zones

### Cosmos.Desktop
Purpose: run an interactive application with visualization and user controls.

Responsibilities:
- load universe and style configuration from JSON
- instantiate and drive the simulation loop
- capture user input and manage simulation state
- render bodies, trails, HUD, and camera motion using Raylib
- expose a simple spacecraft control interface

### Cosmos.Engine.Tests
Purpose: automated test coverage.

Current state: empty skeleton, not wired to the engine code.

### Docs
Purpose: concept and research documentation.

Current state:
- `Docs/Physics/` contains some physics concept notes
- `Docs/Roadmap/` contains planning notes
- `Docs/Architecture/` files are present but empty

## 4. Domain Types

### Entities
- `Body`
  - central domain entity representing mass, position, velocity, acceleration, name, and type
  - mutable position and velocity
- `Spacecraft`
  - sealed subtype of `Body`
  - used for controllable craft in the simulation
- `Universe`
  - container for bodies
  - offers body addition and lookup by name

### Value Objects
- `Mass`
  - encapsulates mass value with validation (> 0, non-infinite, numeric)
- `ReferenceContext`
  - describes a reference frame and optional origin body

### Structs / Records
- `Vector2D`
  - 2D vector value type with operations
- `Vector3D`
  - 3D vector value type with arithmetic, normalization, distance, magnitude

### Orbital Mechanics
- `Orbit`
  - lightweight immutable orbital parameters: apoapsis, periapsis, semi-major axis, eccentricity, period

### Navigation
- `INavigationTarget`
  - navigation target abstraction with `Name`, `Position`, `Velocity`, and `ReferenceContext`
- `BodyNavigationTarget`
  - stub class containing an internal `Body` field only; currently unimplemented

### Enums
- `BodyType`
  - `Star`, `Planet`, `Moon`, `Asteroid`, `Spacecraft`
- `ReferenceFrame`
  - `Inertial`, `BodyCentered`

## 5. Engine Types

### Contracts
- `IIntegrator`
  - integrates body state over time using acceleration and dt
- `IPhysicsModel`
  - advances a `Universe` by a time step

### Services
- `NewtonianPhysicsModel`
  - implements `IPhysicsModel`
  - computes pairwise gravitational acceleration using Newton's law
  - delegates state updates to `IIntegrator`

### Integrators
- `SemiImplicitEulerIntegrator`
  - implements `IIntegrator`
  - updates velocity first, then position
  - applies hard limits:
    - maximum speed
    - maximum distance from origin
  - sets the body’s current acceleration

### Calculators
- `CircularOrbitCalculator`
  - computes circular orbit velocity from central mass and radius
- `EscapeVelocityCalculator`
  - computes escape velocity using `sqrt(2GM/r)`
- `HohmannTransferCalculator`
  - computes delta-v and transfer time for a Hohmann transfer between two circular orbits
- `OrbitalEnergyCalculator`
  - computes kinetic, potential, and total energy relative to a central body
- `OrbitalStatisticsCalculator`
  - computes distance, speed, acceleration, periapsis, apoapsis, eccentricity from tracked orbit data
- `SphereOfInfluenceCalculator`
  - computes sphere of influence radius using the standard mass ratio formula
- `SphereOfInfluenceStatisticsCalculator`
  - wraps sphere of influence calculation and reports whether a body is inside the parent’s SOI

### Analysis
- `OrbitAnalyzer`
  - aggregates calculators to build an overall orbit analysis
  - currently chooses the first body in `Universe.Bodies` as the central body
- `OrbitAnalysis`
  - result object containing statistics, energy, influence, circular orbit velocity, and escape velocity

### Maneuvers
- `HohmannTransfer`
  - immutable record carrying delta-v and transfer time
- `ManeuverPlan`
  - immutable record carrying planned burns
- `ManeuverPlanner`
  - creates a `ManeuverPlan` from current and target orbit radii
- `ManeuverExecutor`
  - applies first burn to a body by adding delta-v along current velocity direction
- `ThrusterSystem`
  - applies prograde and retrograde thrust increments to a body's velocity
- `ManeuverExecutionSystem`
  - executes a burn sequence in discrete steps:
    - apply burn 1
    - wait half transfer time
    - apply burn 2

### Navigation
- `GuidanceComputer`
  - intended to solve navigation problems using orbit analysis
  - currently uninitialized `_orbitAnalyzer` dependency
- `NavigationSolution`
  - contains `OrbitAnalysis`, target, distance, and relative speed

### Tracking
- `OrbitalTracker`
  - tracks periapsis and apoapsis relative to the body named `"Sun"`
  - updates extrema across simulation steps

### Models
- `BurnExecutionState`
  - tracks burn step and timer
- `EnergyStatistics`
  - kinetic, potential, total energy
- `OrbitalInfluenceStatistics`
  - parent body name, SOI radius, boolean inside SOI
- `OrbitalStatistics`
  - distance, speed, acceleration, eccentricity, periapsis, apoapsis

## 6. Desktop Types

### Core runtime
- `Program.cs`
  - manual composition root
  - constructs physics model, integrator, tracker, loaders, renderers, and initial spacecraft
  - contains main loop and simulation orchestration

### Input and state
- `SimulationState`
  - stores simulation speed, pause state, camera, selected body index, current plan, burn target, burn state, controlled body
- `InputHandler`
  - reads keyboard and mouse controls
  - updates camera target, orbit camera orientation, simulation speed, pause state, body selection
  - handles spacecraft thrust input
  - contains a non-invoked `TransferPlanet` Hohmann transfer helper

### Rendering
- `UniverseRenderer`
  - draws the 3D scene and body trails
  - builds a Raylib `Camera3D` from `Camera`
  - uses `PlanetStyleProvider` for render style
- `TrailRenderer`
  - renders historical position traces as a line strip
- `HudRenderer`
  - renders a HUD panel with simulation and orbital metrics
  - recomputes orbital statistics each frame for selected target

### Loading and styling
- `UniverseLoader`
  - loads `solar-system.json`

## 8. Execution Flow

### Startup
- application boots from `Cosmos.Desktop/Program.cs`
- constructs engine and desktop components manually
- loads `solar-system.json` into a `Universe`
- loads rendering style config from `body-styles.json`
- creates `UniverseRenderer`, `HudRenderer`, and `OrbitalTracker`
- creates a spacecraft near Earth and adds it to the universe
- initializes camera target and controlled body

### Main loop
Repeated until window closure:
1. `InputHandler.Handle(...)`
   - updates camera, selection, pause, speed, and thrust controls
   - toggles fullscreen and controlled body
2. If not paused:
   - step physics `SimulationSpeed` times:
     - `NewtonianPhysicsModel.Step`
       - compute acceleration for each body from every other body
       - integrate bodies with `SemiImplicitEulerIntegrator`
     - `OrbitalTracker.Update`
       - update periapsis/apoapsis relative to “Sun”
   - if a burn target exists:
     - `ManeuverExecutionSystem.Update`
       - apply first burn, wait half transfer time, apply second burn
3. `UniverseRenderer.Render`
   - update per-body trails
   - draw each body using Raylib primitives
4. `HudRenderer.Render`
   - draw text metrics, energy status, SOI, transfer plan

### Rendering
- world coordinates are scaled by `RenderScale`
- stars are drawn as golden spheres
- spacecraft are rendered as red cubes
- planets/moons use style colors from JSON config
- trail history is rendered as dark gray 3D lines

## 9. Physics Concepts Implemented

### Gravity and motion
- Newtonian universal gravitation: `F = G * m1 * m2 / r^2`
- acceleration computed for each body by summing attractions from all others
- semi-implicit Euler integration used to update velocity and position

### Orbital metrics
- distance, speed, acceleration magnitude
- periapsis and apoapsis from tracked extrema
- eccentricity estimated from apoapsis/periapsis
- circular orbit velocity
- escape velocity
- kinetic and potential energy
- total mechanical energy

### Orbital mechanics
- Hohmann transfer: delta-v for orbital raising/lowering and transfer time
- sphere of influence radius via standard mass ratio formula
- simple prograde/retrograde thrust application

### Navigation
- `GuidanceComputer` is intended for navigation solutions using orbit analysis
- distance and relative velocity computations are supported
- actual navigation target abstraction is present but not fully implemented

## 11. TODOs and Technical Debt

### Visible TODOs
- `Program.cs` contains a TODO comment indicating future refactor
- `InputHandler.TransferPlanet()` is a temporary Hohmann transfer test and is not called

### Identified technical debt
- hard-coded physics constants:
  - `G = 100` in `NewtonianPhysicsModel`, `OrbitalEnergyCalculator`, `EscapeVelocityCalculator`, `CircularOrbitCalculator`
  - `Mu = 10_000_000` in `HohmannTransferCalculator`
- inconsistent units and no unit system enforcement
- Sun/central body assumptions:
  - `OrbitalTracker` identifies Sun by `Name == "Sun"`
  - `OrbitAnalyzer` selects central body as the first body in `Universe.Bodies`
- incomplete navigation abstractions:
  - `INavigationTarget` is defined but not implemented by any body
  - `BodyNavigationTarget` is an empty stub
  - `GuidanceComputer` has an uninitialized `_orbitAnalyzer` field and no constructor
- incomplete or unused mission planning flow:
  - `ManeuverPlanner` exists, but no runtime path uses it
  - `InputHandler.TransferPlanet()` is not wired into the main loop
- simulation assumptions and hacks:
  - `SemiImplicitEulerIntegrator` clamps maximum speed and position magnitude
  - `ManeuverExecutionSystem` applies tangent burn using a 2D tangent vector with z=0
  - `UniverseRenderer` renders spacecraft with a hard-coded cube instead of unified style
  - `StyleLoader` throws an unprofessional exception message
- inconsistent namespace usage:
  - `UniverseRenderer` defined without namespace, while other desktop types use namespaces
- test infrastructure:
  - `Cosmos.Engine.Tests` contains only `UnitTest1` and no engine references

## 13. Conclusions

Cosmos Engine is organized as a layered simulation project:
- `Cosmos.Domain` defines the core celestial model,
- `Cosmos.Engine` provides physics and orbital computation services,
- `Cosmos.Desktop` provides interactive rendering and user control.

The current codebase is a strong prototype for educational orbital mechanics, but it contains notable implementation assumptions and incomplete components:
- uninitialized navigation plumbing,
- hard-coded central-body assumptions,
- inconsistent namespace and styling practices,
- incomplete test coverage,
- partial documentation coverage.

The next architectural improvements should focus on:
- stabilizing the central-body and coordinate assumptions,
- completing navigation target and guidance abstractions,
- wiring mission planning into the runtime loop,
- cleaning the bootstrap from `Program.cs` into a more modular composition layer,
- improving documentation in `Docs/Architecture/` and adding a developer guide.
